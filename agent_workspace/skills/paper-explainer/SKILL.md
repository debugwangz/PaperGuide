# Paper Explainer

将一篇学术论文转化为零基础也能理解的详细解读文档。

## 核心理念

**目标读者**：对该领域零基础，但有基本数学/编程素养的人

**核心原则**：
1. **逻辑链条清晰** — 每个概念的引入都要有动机，读者要知道"为什么要讲这个"
2. **从 Why 到 What 到 How** — 先讲为什么需要，再讲是什么，最后讲怎么做
3. **关键知识点重点讲** — 识别核心概念，讲透；次要概念一句话带过
4. **补充论文没写的** — 搜索相关资料，填补知识空白

### ⚠️ 逻辑链条规则（最重要！）

**每引入一个新概念，必须先回答：**
> "为什么要讲这个？它解决什么问题？"

**错误示范（❌ 没有动机）：**
```markdown
## 3. 什么是速度场？
速度场是一个函数 v(x,t)，输入位置和时间，输出变化方向...
```
问题：读者会想"为什么突然讲速度场？跟前面有什么关系？"

**正确示范（✅ 先建立动机）：**
```markdown
## 3.1 我们想做什么？
目标：让 AI 从噪声生成图片。

## 3.2 核心问题：怎么从噪声"变成"图片？
需要一点一点变，但每一步该怎么变？

## 3.3 引入速度场：描述"怎么变"
为了描述每一步怎么变，我们需要一个东西告诉我们方向和大小。
这个东西就叫速度场...
```

**逻辑链条模板**：
```
1. 我们想做什么？（目标）
2. 遇到什么问题？（困难）
3. 怎么解决？（引入新概念的动机）
4. 这个概念是什么？（定义）
5. 具体怎么用？（方法）
```

### ⚠️ 不要照搬论文的写法！

**论文作者假设读者是领域专家，但你的读者是零基础！**

| 论文写法 | 问题 | 正确写法 |
|---------|------|---------|
| "GRPO 不需要 Critic 网络" | 什么是 Critic？为什么要 Critic？ | 先解释传统方法为什么需要 Critic，再说 GRPO 怎么避免 |
| "使用 KL 散度正则化" | 什么是 KL 散度？为什么要正则化？ | 先解释问题（reward hacking），再引入 KL 作为解决方案 |
| "基于 Rectified Flow" | 什么是 Rectified Flow？ | 先从"怎么定义中间状态"讲起 |
| "将 ODE 转换为等价 SDE" | 什么是 ODE？什么是 SDE？为什么要转换？ | 先解释 ODE/SDE 的区别（用类比），再解释为什么需要转换 |

**原则**：
- 论文是给专家看的，你的文档是给初学者看的
- 每个术语第一次出现时，都要解释
- 不是翻译论文，是重新组织知识

### ⚠️ 数学公式的处理原则

**不要直接甩公式！** 公式应该是"锦上添花"，不是"必须理解"。

**正确做法**：
1. **先用文字/伪代码解释思路** — 让读者理解在做什么
2. **再给出公式（标注"可跳过"）** — 给想深入的人看
3. **公式后解释每个符号** — 不要假设读者知道
4. **强调"不需要记住公式"** — 降低读者心理压力

### ⛔ 严禁使用的 LaTeX 语法（Streamlit markdown 无法渲染！）

只允许使用 `$...$`（行内）和 `$$...$$`（独立块），其他任何 LaTeX 语法都会显示为乱码！

| 禁止使用 | 正确替代 |
|---------|---------|
| `\(变量\)` | `$变量$` |
| `\[公式\]` | `$$公式$$` |
| `\begin{equation}...\end{equation}` | `$$...$$` |
| `\begin{align}...\end{align}` | 多个独立的 `$$...$$` 块 |
| `\begin{cases}...\end{cases}` | 用文字列举条件（见下方示例） |

**分段函数/条件公式的正确写法**：

```markdown
❌ 错误（LaTeX 环境，Streamlit 无法渲染）：
$$M_{ij} = \begin{cases} 1 & i=j \\ 0 & i \neq j \end{cases}$$

✅ 正确方式一（用文字描述）：
$M_{ij}$ 的规则：
- 当 $i = j$ 时，$M_{ij} = 1$（对角线元素）
- 其他情况，$M_{ij} = 0$（非对角线元素）

✅ 正确方式二（分开写）：
对角线元素：$$M_{ij} = 1 \quad \text{当} \ i = j$$
其他元素：$$M_{ij} = 0 \quad \text{当} \ i \neq j$$
```

### ⚠️ 代码的处理原则

**先不要引入真实代码实现，用伪代码帮助理解！**

**为什么？**
- 真实代码有很多工程细节，会分散注意力
- 伪代码只保留核心逻辑，更容易理解
- 读者先理解思路，再看代码才有意义

**伪代码 vs 真实代码**：

```markdown
❌ 真实代码（太多细节）：
def sde_sample(model, num_steps=50, a=0.7):
    x = torch.randn(1, 3, 512, 512).to(device)
    dt = 1.0 / num_steps
    for i in range(num_steps):
        t = torch.tensor([1.0 - i * dt]).to(device)
        with torch.no_grad():
            v = model(x, t)
        sigma = a * math.sqrt(t.item() / (1 - t.item() + 1e-8))
        ...

✅ 伪代码（只保留核心逻辑）：
从纯噪声开始
循环 50 步：
    1. 神经网络预测速度
    2. 计算噪声强度
    3. 更新图像 = 当前图像 + 方向 × 步长 + 随机噪声
返回生成的图片
```

**什么时候用真实代码？**
- 在"代码实现"章节，读者已经理解原理后
- 或者用户明确要求复现代码

**示例**：
```markdown
#### 伪代码

SDE 采样过程：
1. 从纯噪声 x 开始
2. 循环 N 步：
   - 用神经网络预测当前的"前进方向"
   - 加入随机扰动（这是 SDE 和 ODE 的区别！）
   - 更新 x
3. 返回最终的图片

#### 数学公式（可跳过）
...
```

### ⚠️ 关键知识点识别规则（重要！）

**不是所有术语都要长篇解释，而是识别出关键知识点，重点讲透。**

#### 什么是"关键知识点"？

1. **论文的核心创新依赖它** — 不懂这个就看不懂论文主要贡献
2. **后续内容反复用到它** — 出现频率高，是理解的基础
3. **容易误解或混淆** — 看起来简单但有坑

#### 如何判断？

问自己：
> "如果读者不理解这个概念，他能理解论文的核心贡献吗？"
> - 如果不能 → **关键知识点，重点解释**
> - 如果能 → 一句话带过或脚注

#### 示例：Flow-GRPO 论文

| 概念 | 是否关键 | 理由 |
|------|---------|------|
| 速度场 | ✅ 关键 | Flow Matching 的核心，不懂就看不懂整篇 |
| ODE vs SDE | ✅ 关键 | 论文核心创新就是 ODE→SDE 转换 |
| GRPO | ✅ 关键 | 论文用的 RL 算法 |
| KL 散度 | ⚠️ 中等 | 用到但不是核心，简单解释即可 |
| Euler-Maruyama | ❌ 次要 | 只是离散化方法，一句话带过 |
| Adam 优化器 | ❌ 次要 | 常识，不用解释 |

#### 关键知识点的解释模板

```markdown
### [概念名]

**一句话**：[最简单的定义，不用任何其他术语]

**类比**：[生活中的例子，让人秒懂]

**为什么重要**：[在本文中的作用，不懂会怎样]

**正式定义**：[数学或技术定义]

**常见误区**：[容易搞错的地方]（如果有）
```

#### 次要概念的处理

一句话 + 脚注或链接：
```markdown
使用 Euler-Maruyama 方法离散化（一种常用的 SDE 数值求解方法）。
```

## 工作流程

### 写作原则：先完成，再完善

**不要试图一次写完美！**

```
正确流程：
1. 先写完初稿（允许有问题）
2. 用 read 重新读一遍
3. 发现问题，修改
4. 再读一遍，再改
5. 直到没有明显问题

错误流程：
1. 试图一次写出完美文档
2. 写到一半卡住
3. 反复修改同一段
4. 永远写不完
```

**每次写完一个章节后，立即回头检查这个章节，不要等全部写完。**

### 引用来源（重要！）

**所有非原创内容都要标注来源：**
- 搜索到的博客、教程
- 参考的论文
- 引用的公式推导
- 借鉴的代码

**文中引用格式：**
```markdown
根据 [Lilian Weng 的博客](https://lilianweng.github.io/posts/...)，Flow Matching 的核心思想是...

这个推导参考了 [Liu et al. 2022](https://arxiv.org/abs/...) 的 Rectified Flow 论文。
```

**结尾参考资料格式：**
```markdown
---

## 参考资料

### 论文
- Liu et al., "[Flow Straight and Fast](https://arxiv.org/abs/2209.03003)", arXiv 2022 — Rectified Flow 原论文
- DeepSeek, "[DeepSeekMath](https://arxiv.org/abs/2402.03300)", arXiv 2024 — GRPO 算法来源

### 博客 & 教程

### 代码

```

**为什么要引用？**
1. 尊重原作者
2. 读者可以深入阅读
3. 增加可信度
4. 方便自己回溯

### 第一步：读取论文

1. 用 `read` 读取论文 PDF（如果是 PDF，先用 `pdftotext` 转换）
2. 提取关键信息：
   - 论文标题、作者、机构
   - Abstract（核心贡献）
   - Introduction（问题背景）
   - Method（核心方法）
   - Experiments（实验设置和结果）

### 第二步：识别知识缺口

列出论文中出现的所有：
- 专业术语（如 "Flow Matching", "GRPO", "KL Divergence"）
- 引用的前置工作（如 "based on Rectified Flow [Liu et al.]"）
- 数学符号和公式
- 假设读者已知的概念

### 第三步：搜索补充资料

对每个知识缺口，搜索解释资料：

```
搜索策略：
1. 概念解释：[术语] explained / tutorial / introduction
2. 数学推导：[术语] derivation / proof / math
3. 代码实现：[术语] implementation / code / pytorch
4. 科普文章：[术语] blog / 知乎 / 机器之心
5. 相关论文：[术语] paper / survey
6. 视频讲解：[术语] youtube / bilibili 讲解
```

**优质来源推荐：**
- Lilian Weng's Blog (lilianweng.github.io) — 深度学习综述

### 第四步：构建知识图谱

画出概念依赖关系：

```
目标论文

│   └── B 依赖 B1
```

**从叶子节点开始解释，逐层向上**

## 质量检查

### 自检流程（写完必做！）

**写完后，用 `read` 重新读一遍自己写的文档，逐段检查：**

```
自检清单：

1. 逻辑链条（最重要！）
   - [ ] 每个概念引入前，是否有动机？（为什么要讲这个）
   - [ ] 读者能否回答"这一段和上一段什么关系"？
   - [ ] 是否有逻辑跳跃？（A→C，但没讲 B）
   - [ ] 是否有逻辑冲突？（前后矛盾）
   - [ ] 是否有未解释的假设？（突然假设读者知道某事）

2. 关键知识点
   - [ ] 是否识别出了关键知识点？
   - [ ] 关键知识点是否讲透了？（一句话定义 + 类比 + 为什么重要）
   - [ ] 次要概念是否简洁带过？

3. 零基础友好
   - [ ] 第一次出现的术语是否有解释？
   - [ ] 数学公式是否有直觉解释？
   - [ ] 是否有代码示例帮助理解？

4. 完整性
   - [ ] 是否补充了论文没写的内容？
   - [ ] 是否有扩展阅读资源？
```

### ⚠️ 逻辑断层检查（重点！）

**常见逻辑断层模式：**

| 模式 | 错误示例 | 问题 |
|------|---------|------|
| 跳过动机 | "ODE 是确定性的，RL 需要探索，所以是问题" | 为什么要用 RL？没说！ |
| 跳过中间步骤 | "A 所以 C" | B 在哪里？ |
| 假设读者知道 | "众所周知，X 是..." | 读者不知道！ |
| 前后矛盾 | 前面说 A，后面又说 非A | 自相矛盾 |

**检查方法：**

读每一段时问自己：
> "如果我是零基础读者，读到这里会不会想：等等，为什么？"

如果会，就是有逻辑断层，需要补充。

### 自检示例

读到这一段时：
```markdown
### 3.3 引入速度场：描述"怎么变"
为了描述"每一步怎么变"，我们需要一个东西...
```

问自己：
- ✅ 动机有了吗？→ 有，"为了描述每一步怎么变"
- ✅ 和上一段的关系？→ 上一段问"怎么变"，这一段回答
- ✅ 术语解释了吗？→ 需要检查"速度场"是否解释清楚

### 发现问题怎么办？

如果自检发现问题，**立即修改**，不要等用户指出。

常见问题和修复：
| 问题 | 修复方法 |
|------|---------|
| 概念突然出现 | 在前面加一段动机 |
| 术语没解释 | 加"一句话 + 类比" |
| 逻辑跳跃 | 加过渡段落 |
| 太啰嗦 | 精简次要内容 |
